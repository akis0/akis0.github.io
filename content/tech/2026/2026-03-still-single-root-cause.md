# Root Cause という名の議論停止


## 障害分析の失敗は、組織の認識論の失敗である


障害が起きたとき、組織はしばしば「根本原因は何か」を問う。いかにも真面目で、いかにも合理的に聞こえる。しかし現実には、この問いが原因究明の入口ではなく、思考停止の入口になっていることがある。複数の要因が絡み合って起きた事象に対して、単数形の “root cause” を一つだけ置こうとする。その瞬間、複雑だった現実は、都合のよい一本線に圧縮される。


ここで問題なのは、単に分析が雑だということではない。もっと深刻なのは、その雑な分析が、分析として通用してしまうことである。本来なら、予防に関わる要因、検知を遅らせた要因、影響を拡大した要因、復旧を遅らせた要因、さらには優先順位や責任分界の設計に関わる要因まで、少なくとも複数の contributing factors として整理されるべきなのに、実際には「検知機構がなかった」「確認が不足していた」「コミュニケーションが足りなかった」といった、原因とも対策ともつかない道徳文が “RCA” の名で流通する。もはや、原因分析は技術ではなく儀式である。


だが、さらに厄介なのは、こうした粗い文書がたいてい悪意から生まれるわけではないことだ。人はしばしば、自分のチームの中で安全に言えることしか書けない。開発は実装を見る。運用は設定を見る。QA はテストを見る。PM は進行を見る。それぞれの視界は部分的であり、しかも組織には政治的に言いにくい領域がある。その結果、因果は現実に沿って切られるのではなく、組織図に沿って切られる。多くの障害は複数職能に起因するのに、分析だけがチーム境界で止まる。


だから本当に問うべきなのは、「今回の root cause は何か」だけではない。むしろ、「なぜ私たちは、単一原因を探す古い様式にいまだ依存し、しかもその水準の低い分析を止める仕組みを持っていないのか」である。問題は個々の障害だけではない。問題は、障害から学ぶ能力そのものが、組織の中で劣化していることにある。

ここでいう「古い」とは、単に昔から使われているという意味ではない。事故分析や安全研究の分野では、少なくとも2000年代初頭には、ヒューマンエラーや個別要因を事故の原因として局所化する見方はすでに “old view” と対置され、エラーをより深いシステム条件の症状として捉える “new view” が提示されていた。さらに2005年には、相互に関係する contributing factors や latent organisational factors を扱う systemic occurrence analysis の実務手法も公表されている。2011年以降も、主要事故に単一の root cause を当てる従来型の因果モデルは不十分であり、よりシステム論的な説明が必要だという指摘は繰り返されてきた。したがって、いまなお単線的な root cause の様式に依存しているなら、それは単に言い方が古いのではない。すでに長く乗り越えられるべきだった方法論に留まっているという意味で古いのである。

本稿では、root cause への執着がいかにして組織の学習を歪めるのか、なぜ因果はチーム境界で切ってはならないのか、そしてなぜザルな RCA を書く個人以上に、それを受理しスルーする仕組みのほうが深刻なのかを論じたい。障害分析の失敗とは、障害そのものの失敗である前に、組織の認識論の失敗なのである。


## root cause を root causes にしても足りない


この点について、よくある反論がある。単一の root cause を求めるのが雑だというなら、複数の **root causes** を挙げればよいではないか、という反論である。たしかに、単数形より複数形のほうが現実に近づいたようには見える。一本の棒だった因果が、少しだけ枝分かれしたようにも見える。しかし、この修正は、かなりの部分で問題の核心を温存したまま、表記だけを更新しているにすぎない。


なぜなら**root** という比喩そのものが、すでに現実を誤って切り取っているからである。根という語は、目に見える現象の下に、それを支える本質的な何かが埋まっているというイメージを呼び込む。しかもそのイメージはなお「現象の背後には、それ自体として取り出せる原因が誰かの持ち場の中にあって、分析とはそれを発見することだ」という発想に留まっている。

しかし、複雑な障害はたいてい、そのようにはできていない。そこにあるのは、地下に眠る真の原因ではなく、複数の条件、判断、前提、制約、偶然、設計、運用、観測、組織的配慮が、ある時点で特定のかたちに整列してしまったという事態である。言い換えれば、問題は「根」ではなく**配置**にある。どの要素が存在したかだけではなく、それらがどう組み合わさり、どう連鎖し、どこで止まらず、どこで補足されず、どこで影響が増幅されたのかが重要なのである。


この意味で、root cause を root causes に言い換えるだけでは、単一の雑な原因を一つ出す代わりに、雑な原因を三つか四つ並べるようになるだけである。

見るべきなのは、より素朴だが、より実務的な問いである。**何が起きたのか。何がそれを可能にしたのか。何がそれを止められなかったのか。何が検知を遅らせたのか。何が影響を拡大したのか。何が復旧を遅らせたのか。どの意思決定や優先順位が、その配置を支えたのか。** 目的は、**次に同じ類型の失敗が起きる確率を下げるために、どこに介入すべきかを見つけること**である。そのために必要なのは 介入可能な contributing factors の地図である。地図はときに不格好で、枝分かれしており、しかも枝同士がメッシュ状に絡み合っている。しかし、それが現実である。


つまり、問題は、**そもそも障害を root という語で理解しようとする習慣そのもの**にある。そこから出発する限り、分析はどうしても、単一原因の神話の世界観を抜け出せない。必要なのは、原因論の宗教そのものをやめることである。


障害から学ぶとは、もっと地味な仕事だ。見栄えのする根を掘り当てることではない。複数の要因がどのように重なり、どの境界で見落とされ、どの判断で固定化され、どの防御で止まらなかったのかを、少しずつ記述可能にしていくことである。そして、その記述がチーム境界や政治的配慮によって不自然に切られるなら、そのこと自体がすでに分析対象とすべきである。

だから問うべきなのは、「今回の root causes は何か」ではない。むしろこう問うべきである。**なぜ私たちは、いまだに障害を “root” の言葉でしか記述できないのか。**


## なぜザルな RCA が生まれるのか


では、なぜそのようなザルな RCA が繰り返し生まれるのか。理由の一つは、分析が現実の因果構造に従って行われるのではなく、**発話可能性**に従って行われるからである。人は、自分が見たものを書く。もう少し正確に言えば、自分が見たもののうち、書いても安全なものを書く。開発はコードに視線を固定しやすい。運用は設定や自動化に視線を固定しやすい。QA はテスト観点に、PM は進行や調整に視線を固定しやすい。それ自体は自然である。問題は、その偏りを補正する仕組みがないまま、各自の部分視界を「原因分析」として提出させていることにある。


しかも、複雑な障害ほど、重要な要因は各チームの内部のみならず境界面にも現れる。仕様の解釈、責任分界、優先順位、リスク受容、暗黙の前提、引き継ぎの手触り。こうしたものは、たいてい誰か一人の所有物ではないために書きにくく、分析はチーム内で安全に完結する説明に終始する。こうして、本来は複数職能の関係で起きた失敗が、ある一つのチームのミスか、せいぜいそのチームの改善課題に見えるかたちへと翻訳される。因果は組織図に従わないのに、報告書だけが組織図に従うのである。


ここには、もう一つ別の力学もある。組織はしばしば、分析そのものよりも、**分析が存在すること**を求める。つまり、何が起きたかを丁寧に記述することより、「原因分析をやりました」「再発防止策を出しました」と言える状態を先に欲しがる。すると文書は、探究の記録ではなく、説明責任を形式上満たすための容器になる。その場合、最も重要なのは正確さではない。終わらせやすさである。曖昧で、誰かを過度に刺激せず、しかもそれっぽく見える言葉が好まれる。「確認不足」「コミュニケーション不足」「検知機構の不在」といった表現が好まれるのは、そのためである。しかしこれらの言葉は、発生要因、検知失敗、影響拡大、管理上の不備、対策方針が一緒くたになった、玉虫色の言葉である。単なる巧言であり、再発防止には役に立たない。


だから、ザルな RCA を書く個人を見て「分析能力が低い」とだけ言うのは、問題を表していない。単にその人は**その組織が許容する範囲で最も無難な分析文体**を実践しているだけかもしれない。むしろ問題は執筆者の知性よりも、組織が何をRCAと見なし、何を見なさないかという規範にある。確かに雑な分析は無知から生まれ得る。だがそれ以上に、制度に適応した結果である。


## なぜザルな RCA がスルーされるのか


そして、本当に深刻なのはここからである。ザルな RCA が生まれること自体も問題だが、それ以上に問題なのは、**それがスルーされること**である。

本来、RCA が提出されるということは、それが分析として最低限の要件を満たしているか、どこかで誰かが見ているということを意味するはずである。発生要因と検知要因は分けられているか。影響拡大と復旧遅延は区別されているか。因果の記述と対策案が混同されていないか。チーム境界で不自然に分析が途切れていないか。少なくとも、こうした観点のいくつかはチェックされなければならない。ところが現実には、そこに gate が存在しないことがある。文書は提出され、会議に載り、誰もがやや違和感を抱きつつ、しかし特に止めず、次の議題へ進む。これは些細な運用不備ではない。**分析品質の統治不在**である。

スルーされる理由もまた、個人の怠慢ではなく構造である。第一に、レビューする側がそもそも分析手法を持っていないことがある。分析の粗さを「なんとなく変だ」と感じることはできても、何がどう変なのかを、予防・検知・緩和・復旧、あるいは contributing factors と control points の言葉で言い表せない。その結果、違和感は霧散し、文書だけが残る。第二に、レビュー役が実質的には会議の進行管理者になっていることがある。本来なら分析の quality を管理すべきなのに、実際には schedule と空気の管理に最適化される。第三に、政治的に言いにくい論点に触れたくないという合理的な遠慮がある。チーム外、特に上位の意思決定や優先順位づけに論点が及ぶなら、その不快さを引き受ける人が必要になる。誰もそれを引き受けないなら、分析は言いやすい範囲を動き回って終わる。

だが、これらの事情は「仕方がない」で済む話ではない。もし複数職能で一つのシステムを動かしているなら、そしてその障害の寄与要因がチームの境界を超え得るなら、**チームを超えた問題を安全に記述できる仕組み**が最初から必要なのである。言及しにくいなら、それを可能にするテンプレート、レビュー基準、匿名化された論点化、マネジメントによる明確な後ろ盾が要る。そこまで含めて障害分析のプロセスであって、個々人の勇気に依存してよいものではない。もしチーム外の問題に言及できないなら、それはコミュニケーションの問題ではなく、分析プロセスの欠陥である。

この点から考えれば、blame の向きも変わる。ザルな RCA を書いた現場個人を責めても、あまり意味はない。もちろん、もう少しマシなものを書けただろうという感情は自然である。しかしそれより先に問うべきなのは、**なぜその水準の文書が RCA として提出可能であり、受理可能であり、会議の土台になりえたのか**である。そこにこそ上級管理職層の責任がある。現場が政治的に安全な範囲でしか書けないことは、ある意味で予測可能である。だからこそ管理職には、その制約を見越してなお分析品質が崩れない仕組みを作る責務がある。最低基準を定義すること。不十分な分析を差し戻すこと。境界面やチームを超えた論点を落とさない構造を作ること。古びた単一原因モデルを標準手続から退けること。これらは現場の善意ではなく、管理職の仕事である。

要するに、ザルな RCA の問題は二重である。一つは、その内容が現実の複雑さを記述できていないこと。もう一つは、そのような文書が分析として通ってしまうこと。前者は認識の問題であり、後者は制度の問題である。そして多くの組織では、前者だけが話題にされ、後者は見逃される。しかし本当に深い病変は、後者にある。

## 世界は非線形であり、分析も非線形でなければならない

では、何を変えるべきか。出発点は単純だ。何らかの根本原因がありそれを直せばなんとかなるという線形的世界観から、**世界は非線形だ**という世界観への転換である。私たちが日々扱っているシステムは、単純な部品の足し算ではなく、相互作用する複数の要素からなる複雑なシステムである。小さな前提のずれが予想外に大きな障害へ育つことがあるし、逆に一見大きな変更が何事もなく吸収されることもある。ある要素の不備が別の要素の不備と噛み合ったときだけ破綻が顕在化することもある。重要なのは、個々の要素が単体でどれだけ危険かではなく、**どのような関係の中に置かれたとき、どのように連鎖するか**である。

このような対象に対して、一本の因果線を引いて「これが根本原因だ」と言うのは、便利ではあっても正確ではない。必要なのは、因果を単一の根に還元することではなく、**複数の contributing factors がどのように配置され、相互作用し、失敗を可能にしたかを捉えること**である。

ここでいう contributing factors は、単に「原因をたくさん並べましょう」という話ではない。少なくとも次のような区別が必要である。何が発生に寄与したのか。何が発生を止められなかったのか。何が検知を遅らせたのか。何が影響を拡大したのか。何が復旧を遅らせたのか。どの前提や優先順位や責任分界が、その配置を支えたのか。これらは同じではないし、同じ欄に押し込めてよいものでもない。

この区別は、分析を几帳面に見せるための分類遊びではない。再発防止において、**どこに介入すれば何が変わるか**を考えるために必要である。発生要因に手を入れるのか、検知の遅れを縮めるのか、影響範囲の拡大を防ぐのか、復旧を早めるのか。あるいはそれらの背景にある優先順位や承認フローやチーム間契約を見直すのか。これらは対策の種類も責任主体も違う。にもかかわらず root cause という言葉を使うと、それらのうち一つを除いては無視され、選ばれた一つが最も有効だという保証もない。そればかりか、「確認を強化する」「コミュニケーションを改善する」といった、どこにも着地しない一般論になりやすい。

複雑なシステムとして考えるとき、障害分析の目的は「最も深い原因」を見つけることではない。目的は、**失敗が成立するために必要だった条件の束と、その束の中で介入可能な点を見つけること**へと移る。この観点から contributing factors という考え方が有用である。ある変更もしくは無変更が危険だったのではなく、その変更が曖昧な責任分界、弱い検知、過密な優先順位、暗黙の前提と組み合わさったから危険になったのかもしれない。ある設定ミスが問題だったのではなく、その設定ミスが影響を局所化できないアーキテクチャ、レビューが形骸化した運用、境界条件の不十分な理解と重なったから障害になったのかもしれない。単体では致命的でない要素が、配置の中で致命的になる。ここを見ない限り、次の障害では別の見かけをした同型の失敗をまた迎えることになる。

そして、この見方を採ると、分析の単位も必然的に変わる。複数の役割が協力して一つのサービスを運営している以上、contributing factors はほぼ必然的にチーム境界をまたぐ。設計と実装、実装と運用、運用と監視、監視とインシデント対応、優先順位と品質基準、セキュリティと可用性。こうした境界に生じる摩擦や空白や暗黙知の断絶こそが、しばしば障害の温床になる。したがって、**チームを超えた問題解決こそ必要だ**というのは、協調の美徳を称えるスローガンではない。因果の構造がそうなっている以上、分析と対策もまたそこまで届かざるをえない、というだけの話である。

もちろん、だからといってビッグバンまで遡っても意味はない。どこで分析を止めるかは、組織図ではなく **介入可能性** で決めるべきである。これ以上遡っても、新しい制御点も新しい意思決定も改善余地も出てこない。その時点ではじめて止めればよい。逆に、他チームの優先順位、マネジメントの判断、全社的な基準不在にまで遡ることで初めて介入点が見えるなら、そこまで行くのが自然である。チーム外に出ること自体を禁忌にしてしまった瞬間、分析はすでに現実の記述ではなく配慮の記述になる。

だから変えるべきものは、障害報告書のフォーマットだけではない。分析の背後にある認識論と統治を変えなければならない。第一に、単一原因への執着を捨てること。第二に、障害を複雑なシステムのふるまいとして理解すること。第三に、contributing factors を発生、検知、影響拡大、復旧、組織的条件といった観点から記述すること。第四に、チーム境界で因果探索を止めないこと。第五に、そのような分析がきちんと行われるよう、基準とレビューの仕組みを上級管理職が整えること。ここまで行ってはじめて、blameless という言葉も空疎な慈悲ではなく、学習のための構造になる。

言い換えれば、私たちに必要なのは、根を掘ることではない。**配置を見ること**である。どの要素があり、どの要素がなく、どの境界で断絶が生じ、どの時点で止める機会を失い、どの判断がそれを固定化したのか。その配置を記述し、介入可能な点を見つけ、次回の確率分布を少しでも変えること。障害分析とは、本来そういう地味で実務的な仕事のはずである。

## 結論


結局のところ、問題は「よい root cause を見つけられていない」ことではない。もっと深い問題は、**障害を理解するための言葉そのものが古びており、その古びた言葉がいまだに分析の標準手続として生き残っていること**にある。

単一の root cause を探すという発想は、世界をきれいにしてくれる。責任の所在を見やすくし、会議を閉じやすくし、再発防止策らしきものを箇条書きにしやすくする。しかし、複雑なシステムにおいて起きる障害は、そのような単純化に従ってはくれない。実際に起きているのは、複数の contributing factors が相互作用し、境界面で噛み合い、検知や緩和や復旧の失敗を伴いながら、ある時点で破綻として顕在化するという出来事である。そこに一本の因果線を引いて「これが真因だ」と言い張るなら、それは分析ではなく様式美である。

しかも、雑な RCA の問題は、雑な文書が一つ提出されたことに尽きない。より深刻なのは、そのような文書が RCA として流通し、レビューされ、スルーされ、会議の土台にまでなってしまうことだ。つまり病変は個票の中ではなく、**分析品質のガバナンス**の側にある。

したがって、必要なのは「よりマシな真因」を見つけることではない。必要なのは、障害を複雑なシステムのふるまいとして捉え、contributing factors の配置を記述し、チーム境界を越えて介入点を探すことである。発生に寄与した要因、発生を許した条件、検知を遅らせた要因、影響を拡大した要因、復旧を遅らせた要因、そしてそれらを支えた優先順位や責任分界や基準不在。こうしたものを、単一原因の神話に押し込めるのではなく、そのまま扱えるだけの認識論と制度を持つこと。障害から学ぶとは、本来そういうことである。

もし組織がそれをできないなら、障害分析は学習装置ではなく、責任局所化装置のままであり続けるだろう。Root cause という語はそのたびに使われるだろうが、そこで起きているのは原因究明ではない。原因究明らしさの演出である。会議は閉じる。文書は残る。だが、システムはほとんど賢くならない。

だから最後に問うべきなのは、「今回の障害の根本原因は何か」ではない。**なぜ私たちは、複雑な世界を前にして、いまだに単線的な原因論にしがみついているのか。** 更新されるべきは個別の対策だけではない。**障害をどう読むかという、組織の知性そのもの**である。
